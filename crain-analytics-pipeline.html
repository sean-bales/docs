Crain Website Analytics
How It Works, Where It Breaks

Reader clicks a page. The click becomes a row in a warehouse table that someone else queries to make a business decision. Four layers in between. Five places things go wrong.

Snapshot 2026-04-26

01The pipeline

Reader visits a page. The click becomes a row in a warehouse.

  1. Reader visits a Crain brand page.
  2. Site code fills window.crainDataLayer. Twelve named groups — page, article, user, consent, paywall, more.
  3. Site fires a trigger like analytics.page-article.
  4. Adobe Launch hears it. Runs a rule. Reads data layer fields. Writes Adobe's numbered slots — eVars, props, events.
  5. Adobe stores the hit. Exports hourly to S3.
  6. Lambda loads Redshift. dbt transforms. Tableau, KPI reports, and Crain Answers read the presentation layer.

Three places things go wrong

All three live upstream of the warehouse — in site code or in Adobe Launch. Each one is silent at the moment of edit and surfaces months later in a dashboard.

1 · FIELD-NAME DRIFT · SITE CODE ↔ TAG MANAGER

Site writes one name. Launch reads a different name.

Both layers look healthy in isolation. Slot stays empty in the warehouse.

example: site writes site.iadcmp · Launch reads site.iad_cmp

2 · PER-BRAND SLOT DIVERGENCE · TAG MANAGER

Same field, different Adobe slot per brand. Wired separately, drifted apart.

Cross-brand reports silently lose half the data.

example: site.piano_ab → eVar89 on AMW; eVar136 on Ad Age and Crain's Chicago

3 · DISABLED OR LEFTOVER RULES · TAG MANAGER

Rules turned off per brand. Or marked TEMPORARY and never removed.

example: TEMPORARY bypass Piano paywall rule resident on two brand properties since Sept 2025

WHY THESE PERSIST

Site code and tag manager are owned by different groups. Per-brand wiring is owned by per-brand editors. None of them feel a bug at the moment of edit — drift surfaces months later in a dashboard query, far from the layer that introduced it.

02A closer look

Six steps in detail.

Same path as the diagram, with concrete examples. Walk through this when the room wants to know what each layer actually does — not just the box it sits in.

1Site code fills the data layer

Page renders. Site code populates one object — window.crainDataLayer. A JavaScript object on the page. Browser dev tools can read it. Twelve named groups inside. Each group holds facts about one thing.

GroupWhat's in it
pagepage type, section path, URL
articlearticle id, headline, author, publish date, paywall state
userlogged-in flag, user id, subscription tier, industry
consentwhat the reader consented to (advertising, functional, performance)
wallpaywall outcome — shown, dismissed, converted
sophicontent recommendation decisions
productsubscription product details — only set during checkout
sitecampaign codes, internal test markers
analyticsthe trigger string for the current event
event · content · blueconicsecondary fields — Adobe-event flags, content metadata, CDP profile data

Different pages fill different groups. An article page fills page and article. A checkout page also fills product. A logged-in reader's hits also fill user. By the time the page is ready, the object is full.

2Site fires a trigger

Site dispatches a browser event. Format: analytics.<name>. The event name is the trigger. Two kinds.

Page triggers — fire once per page when the page is ready

TriggerWhen it fires
analytics.page-articlearticle view
analytics.page-sectionsection or homepage view
analytics.page-searchsearch results page
analytics.page-sub5subscription checkout, step 5
analytics.page-sub6subscription completed
analytics.page-logindonelogin completed
…twenty-some others

Click triggers — fire when reader clicks a specific element

TriggerWhen it fires
analytics.click-articlereadreader finished reading article
analytics.click-articlegiftshrreader clicked gift-share
analytics.click-searchmorereader clicked "load more results"
analytics.click-emailexchemail exchange form submitted

When the trigger fires, window.crainDataLayer is already populated. The trigger is the site saying something happened. here's the name.

3Adobe Launch hears the trigger

Adobe Launch is a tag manager. Browser-side code. Subscribes to every analytics.* event. Trigger fires; Launch hears it; Launch looks up its rules.

A rule is a recipe. One rule, one trigger. Example rule on Ad Age for the analytics.page-article trigger:

read crainDataLayer.article.id          → write to eVar5
read crainDataLayer.article.author      → write to eVar9
read crainDataLayer.user.subscription   → write to eVar72
set prop62 = "page-article"
fire event3

Rule reads named fields out of the data layer. Rule writes those values into Adobe's numbered slots. Roughly 87 rules per brand. Roughly 175 named pointers (data elements) per brand backing those rules. Eight separate Launch properties — one per brand-group.

4Adobe stores the hit

Launch sends a structured row to Adobe. ~400 columns. That's the hit. Adobe Analytics stores it. Slots in three flavors.

FlavorRangeWhat for
eVar1–250persistent values — user id, content type, brand
prop1–75per-hit values — trigger name, page section
event1–100+counters — login completed, purchase

Whichever slots the rule wrote, those slots fill on this hit. Other slots stay empty for this hit. One report suite per brand. Roughly 22 active across Crain.

5Adobe exports to S3, Lambda loads Redshift

Adobe exports hourly. Drops the full row into Crain's S3 bucket — Adobe Data Feed. AWS Lambda picks up the file. COPY into Redshift. Lands in cdw_01_raw.raw_hits. dbt promotes through three layers.

LayerWhat's there
cdw_01_rawraw landing — every hit, every column
cdw_02_stgcleaned, joined, renamed — digital_events is the curated view
cdw_03_prsKPI tables — kpi_dashboard, article_performance

Tableau, KPI reports, Crain Answers, analyst SQL — they all read the presentation layer.

6A second path, used sparingly

Some on-page React hooks bypass Launch. Call Adobe directly. Hardcoded list of which fields go to which slots. About 25 slots. Faster path. But the rule-attribution stamp — prop62, prop63 — never gets set on those hits.

Why some hits in the warehouse have an empty prop62: those are the bypass hits.

03Triage cheat sheet

When a bug lands in your queue.

Four real-shaped examples. Each shows what lands in your queue, how to triage it in thirty seconds, and what the developer-ready ticket looks like. Reporters send symptoms. Translate yourself.

Example 01

"Adobe page tags stopped firing on [brand]"

What's happening

Tags not firing. Adobe is recording nothing. Not a warehouse problem.

Suspect first

Adobe Launch — rule disabled, consent gate stuck — or site-side script-load regression.

Two questions before you escalate
  • Did anything ship near the date? TrustArc, Piano, Sophi, BlueConic, ARC migration, compliance refactor, feature flag. If yes, that's the lead.
  • Whole brand, or one page type? Whole brand → consent gating or property-level rule. One page type → specific rule.
Developer-ready ticket
Brand / RSID: [brand].com / [rsid]
Drop window: hits/hour [N] → [M] starting [date]
Scope: all page types / one page type
Recent changes: [list anything that shipped near the date]
Likely layer: Adobe Launch (rule status, consent gate) OR site-side script load
Evidence: network tab on representative page; Adobe debugger output; change log
Example 02

"Industry / license_id missing for logged-in users on [brand]"

What's happening

Field-name drift between site code and Launch — or site never writes the field. Not a recent regression. Chronic shape.

Suspect first

Site code ↔ tag manager interface.

Diagnostic

Open dev tools on a logged-in test user. Inspect window.crainDataLayer.user.

  • Field present in browser → drift. Compare data layer key vs Launch data element path character-by-character.
  • Field missing in browser → site never writes it for this segment. Engineering owns.
Developer-ready ticket
Brand / RSID: [brand].com / [rsid]
Symptom: logged-in segment shows blank for industry / license_id / supexpire
Browser dump: window.crainDataLayer.user = [paste]
Launch DE path: [paste data element settings]
Likely layer: site code ↔ tag manager (drift) OR site code (field never written)
Evidence: dev tools dump; Launch DE screenshot; Adobe debugger view
Example 03

"eVar70 (article_type) shows coupon codes"

What's happening

Stale warehouse label. The Adobe slot was repurposed in Launch. The warehouse seed kept the old name. Six eVars currently mislabeled this way.

Suspect first

dbt seed. Not Launch, not Adobe.

Confirm in 30 seconds
  • Live wiring: Adobe Analytics | page-sub5 and page-sub6 write crainDataLayer.product.coupon to eVar70.
  • Seed: evar_mapping.csv still labels eVar70 as article_type.
  • Coupon codes are correct content. Column is mislabeled.

Not a data bug. Labeling bug.

Developer-ready ticket
Slot: eVar70 (column: post_evar70_article_type)
Top-5 distinct values last 30d: [list — confirm coupon-shaped]
Live wiring: Launch writes crainDataLayer.product.coupon on subscription rules
Seed says: article_type
Likely layer: dbt seed (cdw-dbt/cdw/seeds/evar_mapping.csv)
Proposed fix: rename column OR new column + deprecate old. Both need consumer migration.
Example 04

"post_evar1 is empty even though Adobe shows the data"

What's happening

Redshift column-type drop. Column is bigint. Adobe sends text. COPY drops the text as NULL. ~28 columns affected.

Suspect first

Redshift load. Not Launch, not Adobe, not site.

Diagnostic in two queries
select count(*) from raw_hits where post_evar1 is not null;  -- 16
select count(*) from raw_hits where post_prop1 is not null;  -- 46.2M

Companion prop carries content. eVar empty. Bigint typing. S3 sources are append-only — content recoverable on re-COPY after type fix.

Developer-ready ticket
Column: post_evar1 (current type: bigint per information_schema.columns)
90-day non-null: 16 rows
Companion post_prop1 (text-typed): 46.2M rows, same content
Likely layer: Redshift COPY into raw_hits
Proposed fix: ALTER column type to varchar; re-COPY from S3; ~27 other columns same shape

Not a bug — common false alarms

  • Empty-string defaults on Adobe slots. Most data elements default to "". "Field not present" looks identical to "field empty." Confirm the field was supposed to be set on this page, flow, or user state.
  • Pre-2025-05-01 step changes on eVars 126+. Schema split that date. Pre-cutoff rows NULL-stuff those columns by design.
  • 78% null on customer account id. Anonymous traffic. Right denominator for join health is non-null hits, not total hits.
  • Three GenomeWeb-related report suites collapse to one warehouse code. Intentional. One Pelcro tenant covers all three.
  • prop62 / prop63 carrying long strings. Design. Those slots are the rule-attribution forensic trail. If they look broken, the bug is shape 5 (bigint drop), not Launch leaking debug data.
