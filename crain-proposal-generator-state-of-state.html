01Why rebuild

Scopic delivered the MVP. Crain moved it in-house.

Current state. Rationale. What the rebuild adds.

  • What Scopic delivered. A working proposal tool. RFP in, draft out, deck export. Smartsheet-driven pricing. Five city brands.
  • Why rebuild. Claude Enterprise landed across Crain. Moving in-house fit the stack. Rebuild scope extends beyond Scopic's footprint.
  • Gains from the rebuild. Higher-quality drafts. Enhanced capabilities — auto-ingestion, structured extraction, broader coverage. Smaller maintenance surface.

Sources: Scopic delivery repos; Claude Enterprise tenant; approved October 2025 specifications; rebuild scoping 04/2026.

02Path forward

Proposal work moves to Claude Enterprise on native connectors.

Directionally viable. Implementation details open.

Panel 1 · Shape

Current
  • Custom web application. Hosted on dedicated cloud infrastructure and maintained in three repositories.
  • AI engine wraps OpenAI. Custom retrieval, custom orchestration, custom chunking.
  • Proposal state lives in a purpose-built database. Dashboard and archive layered on top.
  • Pricing flows on a schedule. Smartsheet refreshes into the database hourly.
  • Media kits arrive via an ingestion script. Run by an operator when content changes.
  • Login uses email plus one-time code. Roles are defined inside the application.
Future — first-pass read
  • Claude Enterprise project. No Crain-built application; behavior lives in a system prompt.
  • Claude reads natively. Hosted Microsoft 365 and Smartsheet connectors, plus one small custom write surface.
  • Proposal state lives in SharePoint. Dashboard and archive are SharePoint list views.
  • Pricing reads live. Smartsheet queried at proposal time, on demand.
  • Media kits read live. SharePoint queried at proposal time, on demand.
  • Login uses Crain single sign-on. Roles are defined in Entra groups.

03Phases

Three phases, very TBD. Pending alignment with ops, sales leadership, Finance, and Crain IT.

High-level only. Detail lives in the appendix.

  • Phase 1 — MVP (TBD). Ship on current Smartsheet and SharePoint. Claude reads native PDFs and pricing rows. Free-text reconciliation hint on every line.
  • Phase 2 — bridge on Naviga read access (TBD). Add structured Naviga foreign keys to every line — Rate Line ID, Rate Card ID, GL Type, Cost Center, Company, plus snapshot date. Reconciliation becomes a lookup. Unlocks when Naviga API read lands or AdOps cross-brand exports are adopted as the read path.
  • Phase 3 — consolidate on a mart (TBD). Analytical layer for cross-proposal reporting, cross-brand rollups, and historical win-loss analysis.

A2Standardization

Standardization runs four layers. The agentic direction absorbs the leaf layer at read time.

  • Layer one — Sales Family. ~12 candidates per canonical data analysis. Sales-defined, not a Naviga concept. Stage-2 ratifies the final list with sales leadership at the table.
  • Layer two — GL Type. 71 codes enterprise-wide. Finance-owned. Registry-settled. Examples: DNEW (Newsletters), CSPC (Custom Sponsored Content), PDIS (Print Display).
  • Layer three — publications. 21 active across 2 Companies (01 CCI, 02 LTD). In Naviga this is Cost Center. Five publications book to LTD (Europe).
  • Layer four — per-product identity. ~4,000–5,000 Rate Lines enterprise-wide. Naviga IS the register — no Crain-side minting. 65% of Rate Line IDs persist year-over-year; 35% reissue. Snapshot date at resolve-time handles reissue.
  • Directional hypothesis. Semantic match at read time for MVP. Phase 2 anchors on Rate Line ID (naviga_rate_line_id + snapshot date) once the read path lands. Phase 3 consolidates on a mart.
Open — discussion Resolution here shapes how much weight the semantic-match hypothesis has to carry versus the structured-FK Phase-2 path.

A3Product data model · stakeholder view

The three-view framing maps onto five entities, plus a sixth for phase-2. Family splits two ways.

Tentative Two things can reshape this: (1) Naviga API access and any schema change that surfaces when it lands; (2) blockers or preferences the stakeholder group raises after review.
  • Brand (1). The publication — Modern Healthcare, Crain's Chicago Business, Ad Age. In Naviga this is encoded as Cost Center. 21 active publications across 2 Companies.
  • Sales Family (2a). How sales talks about the category — Newsletters, Custom Content, Recognition Programs, Event Sponsorship. ~12 candidates; Stage-2 ratifies.
  • GL Type (2b). How Finance books the revenue — 71 codes enterprise-wide. Finance-owned, already settled.
  • Product Offering (3). The sellable unit — one Naviga Rate Line. Where rate, lead time, and creative specs live. Naviga IS the Offering register.
  • Resolved Line Item (4). One row in one proposal's strategy.json — which Offering an AE picked, at what price, for what dates, with Claude's reasoning attached.
  • Execution Record (5). Delivery-side data from specialist tools — Jotform leads, event registrations, ad-server logs. Named now; tool builds this phase-2 (TBD).

Panel 2 · What stays, what changes, what's pending

What stays
  • Cross-publication category work becomes the Sales Family layer — applied at the right layer instead of fighting the ID.
  • Prior single-publication template already anchored on Naviga's GL Type. The full Naviga master confirms the instinct.
  • Cross-brand overlap lands at Sales Family + GL Type, not at a shared product ID.
  • Awards and recognition content (40 Under 40, Notable Women, The Book, Best Places to Work) routes through existing GL Types by format plus optional RC2425 Revenue Category. Surfacing what's already in Naviga, not inventing a bucket.
  • Earlier Naviga mapping work preserves as a structured set of FKs on every Offering, populated from AdOps's quarterly cross-brand exports (23 files already on SharePoint).
What changes
  • Product ID stops doing three jobs. Offering identity = Naviga's Rate Line ID. Sales Family and GL Type are separate tags. Cost Center + Company carry publication/billing scope.
  • Family splits two ways. Sales Family (sales-defined, ~12 candidates, Stage-2 ratifies) and GL Type (Finance-owned, 71 codes, settled). Two views, two cadences.
  • Naviga mapping is not one column. It's a nine-key FK bundle on every Offering plus a snapshot date to handle the 35% YoY Rate Line reissue rate.
  • 2027 rollover handled by Naviga. Ratecard IDs carry year context; Rate Line IDs persist 65% year-over-year; snapshot at resolve-time keeps every past proposal correct. No Smartsheet "never overwrite" rule required.
What's pending (TBD)
  • Naviga API read access. Accelerates Phase 2. AdOps cross-brand exports already cover the schema and work as the interim read path.
  • Stakeholder review of this model before team-wide ratification.
  • Stage-2 ratification output — final Sales Family list and alias map for 19 sub-brand Cost Centers. GL Type list already settled.
  • Four open validation items — CRE2 GL Type origin; 46 Editorial rows dropped during Scopic normalization; RC2425 usage pattern; Smartsheet vs. AdOps operational read path.

A4Product data model · technical view

Six entities model the data. Five ship MVP. Execution Record waits phase-2.

Tentative Two things can reshape this: (1) Naviga API read access and any schema change that surfaces when it lands; (2) blockers or preferences the stakeholder group raises after review.

Offering identity = Naviga's Rate Line ID + snapshot date. Family splits into Sales Family (sales-defined, ~12 candidates) and GL Type (Finance-owned, 71 codes, settled). Every Offering carries a nine-key Naviga FK bundle. naviga_reconciliation_hint demotes to a nullable fallback for custom one-offs; structured FKs are the primary path.

Panel 4 · Tradeoffs

  • Day-one reconciliation stays free-text. Phase-1 MVP carries naviga_reconciliation_hint as primary; Phase-2 extends Line Items with the structured offering_ref FK bundle. Day-one cost is no worse than today; Phase-2 cost drops.
  • Execution Record named but not built. Phase-2 first-class problem after 2c. Naming it now so the team has a word for the Jotform / Excel / ad-server pattern and its analogues.
  • frequency_cap near-zero populated today. No tab in existing working files supplies numeric caps. Constraint rule 8 degrades to no-op per Offering on blanks — correct behavior. Populate over time, not at once.
